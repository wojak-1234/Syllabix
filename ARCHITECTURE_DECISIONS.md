## Phase 1 — 아이디어 발굴 : 공모전 주제 해석

### 배경
공모전 주제: **"AI 활용 차세대 교육 솔루션"**  
심사 기준: 기술적 완성도 / AI 활용 능력 / 기획력 및 실무 접합성 / 창의성

### 초기 질문
> "수상권에 들 법한, 기존 서비스에는 없는 아이디어를 제시해줘"

### AI 제안 (6가지 아이디어)
AI가 제안한 후보군:

| 아이디어 | 설명 |
|---|---|
| AI 강의 미러링 & 질 분석 코치 | 강의 녹화본 분석 → 강사 피드백 |
| AI 학습 공백 레이더 | 행동 데이터로 "모르는 줄 모르는" 개념 탐지 |
| AI 교육 운영 자동화 | 문의·행정 반복 업무 자동화 |
| 실시간 AI 이해도 체크 레이어 | 라이브 강의 중 이해도 실시간 감지 |
| **AI 커리큘럼 역설계 엔진** | 목표에서 커리큘럼을 거꾸로 설계 |
| AI 피어러닝 다이나믹 매칭 | 강점이 다른 학습자를 AI가 매칭 |

### 의사결정 1 — 방향 후보 선택
- **채택**: AI 커리큘럼 역설계 엔진 + AI 학습 공백 레이더 (Blind Point)
- **기각**: 강의 미러링, 운영 자동화, 피어러닝 매칭
- **기각 이유**:
  - 운영 자동화는 교육 솔루션보다 CS 툴에 가까움
  - 피어러닝 매칭은 콘텐츠 없이 매칭만으로는 약함
  - 강의 미러링은 강사 수가 적은 초기 서비스에 데이터 부족

> **핵심 인사이트**: "목표에서 역산하는 커리큘럼"과 "모르는 걸 AI가 먼저 발견"이라는 두 축이 이 프로젝트의 정체성이 됨

## Phase 2 — 기능 정의 : 첫 번째 기획안

### 배경
Phase 1에서 선택한 두 축을 기반으로 구체적인 기능 목록을 정의

### 초기 기획 (사람이 먼저 제시)

**Student Side**
- AI 커스텀 커리큘럼 생성
- Frequent Blunder List (오답노트)
- Code Reviewer
- Blind Point Report & Dynamic Curriculum Reset

**Teacher Side**
- 강의 업로드
- Blind Point 확인
- 학생 실패 포인트 확인

### AI 구현 방안 제안
AI가 각 기능별 구현 방식, 기술 스택, 데이터 흐름을 제안

### 의사결정 2 — 기능별 구현 방식 채택
| 기능 | AI 제안 | 채택 여부 | 이유 |
|---|---|---|---|
| 커리큘럼 생성 | Structured Output + RAG | ✅ 채택 | JSON 강제 반환으로 파싱 오류 없음 |
| 블런더 리스트 | 오답 클러스터링 + Claude 분석 | ✅ 채택 | 행동 데이터 기반 패턴 추출 |
| Code Reviewer | Multi-step Chain (3단계) | ✅ 채택 | 단일 호출보다 맥락 반영 정밀도 높음 |
| Blind Point | LangChain Agent | ✅ 채택 | 자율 도구 호출 구조가 핵심 차별화 |

> **이 단계의 문제**: Teacher 기능이 "보여주기"에 그침. 강사가 Blind Point를 받아도 강의를 수정하기까지 행동 동선이 없음

## Phase 3 — AI 기술 다양화 : Agentic AI 도입

### 배경
> "AI들을 단순 chatbot 형태가 아닌 다른 여러 방식으로 활용하고 싶어"

### AI가 제안한 5가지 패턴
1. **Structured Output** — JSON 생성 엔진
2. **RAG** — 강의 콘텐츠 기반 답변
3. **LangChain Agent** — 자율 판단·행동 (Blind Point 탐지)
4. **Multi-step Chain (LCEL)** — 코드 리뷰 3단 파이프라인
5. **Streaming** — 실시간 피드백 UX

### 의사결정 3 — Agentic AI 구조 채택
- **채택**: Agent (Blind Point 탐지) + Multi-step Chain (코드 리뷰) + Streaming (UX)
- **판단**: 단순 Claude 호출이 아닌, AI가 스스로 도구를 선택해 순서대로 실행하는 구조가 심사 차별화 포인트

**Blind Point Agent 도구 목록 확정**
```
Tool 1: get_behavior_logs       — 행동 로그 조회
Tool 2: get_wrong_answer_patterns — 오답 패턴 분석
Tool 3: update_blunder_book     — 블런더북 업데이트
Tool 4: trigger_curriculum_reset — 커리큘럼 리셋
Tool 5: notify_teacher          — 강사 알림 발송
```

> **핵심 전환점**: "AI가 대답한다"에서 "AI가 판단하고 행동한다"로 개념 전환

## Phase 4 — 방향 전환 1 : 범용 플랫폼 시도

### 배경
> "코딩으로만 주제를 국한하면 좀 그래. 제빵, 위험물관리사 같은 포괄적인 주제를 다룰 수 있게 방향성을 변경한다면?"

### AI 제안 — 범용 플랫폼 메뉴 재설계
- 제거: Code Reviewer, 오답노트 (코딩 전용)
- 추가: AI 개념 탐험가 (Knowledge Graph), AI 모의시험 생성기, 실전 시뮬레이터, AI 학습 저널, 커뮤니티 Q&A

### 의사결정 4 — 범용 전환 시도 후 재검토 예정
- **임시 채택**: 범용 방향 탐색
- **보류 이유**: IDE와 코드 리뷰어를 유지하면 어정쩡해짐. 범용으로 가면 실습 메커니즘 전체를 교체해야 함

> **이 시점의 딜레마**: 넓게 가면 임팩트가 분산되고, 좁게 가면 시장이 작아 보임

## Phase 5 — UX 진단 : 기존 기획의 문제점 발견

### 배경
> "내 UX는 뭐가 문제인 것 같아?"

### 기존 기획 흐름 재검토
**Teacher**: 수요확인 → 강의제작 → 업로드 → Blind Point 수신  
**Student**: AI 커리큘럼 생성 → 문제 제시 → Dynamic IDE

### AI가 발견한 UX 문제 4가지

#### 문제 1 — Teacher: 수요 확인과 강의 제작 사이 단절 `치명적`
- **현상**: 수요를 보여줘도 강의 제작으로 이어지는 동선 없음
- **원인**: 강사는 수요를 알아도 강의를 만드는 게 부담
- **해결**: 수요 키워드 클릭 → AI 강의 초안 원클릭 생성 → 강사는 검토만

#### 문제 2 — Student: 커리큘럼 생성 후 "그래서 뭐 해?" 공백
- **현상**: 로드맵을 받고 나서 첫 강의를 어떻게 찾는지 동선 없음
- **해결**: 커리큘럼 주차 클릭 → 관련 강의 자동 추천 → 퀴즈 → IDE 원스텝

#### 문제 3 — Blind Point가 강사에게만 가고 학생에게 안 감
- **현상**: 강사가 강의를 수정하는 데 시간이 걸리는 동안 학생은 계속 틀림
- **해결**: Blind Point → 강사(강의 개선) + 학생(즉시 보강) 동시 발송

#### 문제 4 — 진단 없이 커리큘럼 생성
- **현상**: 현재 수준 데이터 없이 목표만으로 커리큘럼 생성 → 미스매치 발생
- **해결**: 온보딩 진단 5문제 → 수준 파악 후 커리큘럼 생성

### 의사결정 5 — UX 구조 전면 개편
- **채택**: 4가지 문제 모두 수정
- **추가 도입**: 온보딩 진단 테스트, 학습 루프 닫힘 (취약점 → 자동 복귀 → 재도전)

## Phase 6 — 방향 전환 2 : 코딩 특화 확정

### 배경
> "코딩에만 국한한다면 어떤 기능을 넣으면 좋을까? UX를 매끄럽게 다듬고 싶어"

### AI 제안 — 코딩 특화 추가 기능
- **AI 페어 프로그래머**: 소크라테스식 질문으로 자력 해결 유도
- **시도 히스토리 타임라인**: 코드 제출 과정 저장 → Blind Point 정밀도 향상
- **학습 패턴 뱃지**: 힌트 없이 해결, 클린코드 달성 등
- **AI 다중 풀이 비교**: 버블정렬 성공 후 퀵정렬·머지정렬 비교 제시

### 의사결정 6 — 코딩 특화 최종 확정
- **채택**: 코딩 특화 유지
- **채택 이유**:
  - IDE는 코드 실행 환경이 있어야 의미 있음
  - "답을 주지 않는 AI 페어 프로그래머"가 플랫폼 핵심 가치와 정확히 일치
  - 타겟이 명확할수록 심사 기준 "실무 접합성"에서 유리
- **기각**: 범용 플랫폼 방향
- **기각 이유**: IDE + 코드 리뷰 + 소크라테스 모드가 코딩 없이는 성립 안 됨

**최종 확정 학습 루프**
```
진단 테스트 → AI 커리큘럼 생성 → 강의 추천·수강
→ 개념 확인 퀴즈 → Dynamic IDE (AI 페어 프로그래머)
→ Blind Point 탐지 → 자동 복귀·재도전 → 다음 주차
```

> **결론**: "범용으로 넓히면 임팩트가 분산된다. 코딩 특화로 깊게 파는 게 낫다" — 사람의 최종 판단

## Phase 7 — AI 협업 전략 : Agentic AI + 문서화

### 배경
> "AI와 효율적으로 협업하는 능력이 가장 중요한 심사 항목인 것 같아. 최신 Agentic AI를 사용하는 건 어때?"

### AI의 반응 — 방향 조정 제안
AI가 "Agentic AI 기술 자체보다 협업 과정 문서화가 실제 평가 포인트"라고 짚음

### 의사결정 7 — 두 축 동시 전략 채택
- **전략 A**: AI 협업 과정 체계적 문서화 (GitHub docs/)
- **전략 B**: 플랫폼 내 Agentic AI 기능 실제 구현

**문서화 구조 확정**
```
/docs
  /ai-sessions/         ← 날짜별 협업 세션 로그
  AI_PLANNING.md        ← 기획 의사결정 과정
  ARCHITECTURE_DECISIONS.md ← 기능별 AI 패턴 선택 근거
  PROMPT_GUIDE.md       ← 프롬프트 설계 전략
  AI_REVIEW_LOG.md      ← AI 제안 → 사람 검토 흔적
```

**Agentic AI 적용 우선순위 확정**
1. AI 페어 프로그래머 (Multi-turn Agent) — 데모 핵심
2. Blind Point 탐지 Agent (LangChain) — Agentic의 증거
3. 커리큘럼 생성 (Structured Output + RAG) — 기반 기능

 ## Phase 8 — 구현 및 최적화 : 실시간 AI 페어 프로그래머와 모델 통합
 
 ### 배경
 > "학생 IDE에서 제출 시 실시간으로 오류를 감지하고 힌트를 주는 기능을 구현하고 싶어. 그리고 전체 모델을 최신 버전인 gemini-3.1-flash-lite-preview로 변경해줘."
 
 ### 주요 구현 내용
 1. **Dynamic Code Evaluator (API)**: 학생의 코드와 문제 정보를 Gemini에게 전달하여 문법/논리 오류를 교정하고 방향성을 제시하는 에이전트 구축.
 2. **Socratic Feedback Loop**: 정답을 직접 주지 않고, 시도 횟수에 따라 힌트의 구체성을 조절하는 프롬프트 엔지니어링 적용.
 3. **모델 통합 (Consolidation)**: 모든 AI 티어를 `gemini-3.1-flash-lite-preview`로 단일화하여 지연 시간(Latency) 감소 및 비용 최적화.
 
 ### 의사결정 8 — 실시간 분석 및 모델 전환 채택
 - **채택**: `/api/student/evaluate` 라우트 생성 및 프론트엔드 IDE 연동
 - **채택 이유**:
   - 하드코딩된 정답 체크는 창의적인 풀이를 방해함. AI 분석만이 다양한 정답 로직을 수용 가능.
   - `3.1-flash-lite` 모델은 코드 분석에 충분한 성능을 보이면서도 응답 속도가 매우 빨라 실시간 IDE 환경에 적합함.
 
 > **핵심 성과**: "답을 주지 않는 AI 튜터"라는 플랫폼의 정체성이 실제 작동하는 코드(IDE)로 구현됨.

## 최종 확정된 기획 방향

### 정체성
> "AI가 답을 주지 않는 코딩 교육 플랫폼.  
> 학습자가 스스로 깨닫도록 유도하고,  
> AI가 학습자보다 먼저 막힌 곳을 발견한다."

### 핵심 기능 (확정)
- 온보딩 진단 → AI 커리큘럼 생성 → 강의 추천
- Dynamic IDE + AI 페어 프로그래머 (소크라테스 모드)
- LangChain Blind Point 탐지 Agent
- Teacher Blind Point 대시보드 + AI 강의 초안 생성

### AI 기술 스택 (확정)
- Claude API (claude-sonnet-4-6)
- LangChain (Agent, LCEL Chain)
- RAG (pgvector)
- Structured Output (tool_choice 강제)

### 배경
학습자가 코딩 테스트 중 단순히 "정답 여부"만 아는 것을 넘어, 스스로 논리적 오류를 해결할 수 있도록 돕는 실질적인 학습 도구(Scaffolding)가 필요함.

|---|---|---|

| **Socratic Chatbot** | 3회 실패 후 활성화되는 채팅 | 정답 제공을 지양하고 질문을 통해 간접적인 해결책 유도 |

### AI 프롬프트 프레임워크: TCREI & Harness Engineering
AI 튜터의 역할을 명확히 하고 일관된 교육적 가치를 제공하기 위해 다음 프레임워크를 적용함.
- **TCREI (Title, Context, Role, Execution, Instruction)**: AI에게 명확한 페르소나와 실행 지침 부여.
- **Harness Engineering**: AI가 정답 코드를 절대로 직접 주지 못하도록 출력 구조와 제약 조건을 강력하게 제어(Harnessing).

> **핵심 가치**: "AI는 지시하는 자가 아니라, 함께 고민하는 동료이자 질문하는 스승이다."

## AI 협업에서 배운 것

### AI가 잘한 것
- 기존 서비스에 없는 각도로 아이디어 발굴
- UX 흐름의 단절 지점을 구조적으로 진단

### 사람이 판단한 것
- 범용 vs 코딩 특화 → **코딩 특화** (AI는 둘 다 가능하다 했지만 사람이 결정)
- Agentic AI 도입 → **채택** (기술 트렌드와 심사 기준 교차점 판단)
- 오답노트 제거 → **채택** (AI 페어 프로그래머가 더 강력하다 판단)

### AI 제안을 기각한 이유
- AI는 "범용도 가능하다"고 했지만, 실제 데모에서 코딩 특화가 훨씬 강력함
- AI는 기능을 많이 제안했지만, MVP 범위로 줄이는 건 사람의 판단

  ## Phase 9 — 스택 최적화 및 최신 프레임워크 대응 : Next.js 15+ 전환
  
  ### 배경
  > "Next.js 버전 업그레이드(15+)에 따른 컴포넌트 런타임 에러 발생 및 AI 생성 커리큘럼의 데이터 연결성 강화 필요"
  
  ### 주요 구현 및 수정 내용
  1. **Async Dynamic APIs 대응**: Next.js 15+에서 `params`가 Promise로 변경됨에 따라 모든 동적 라우트 페이지에 `React.use()` 패턴 전면 도입 (Breaking Change 대응).
  2. **AI 모델명 원복 및 고도화**: 사용자 요청에 따라 `gemini-3.1-flash-lite-preview` 모델을 전면 적용하여 최신 모델의 추론 능력 활용.
  3. **Mock 데이터의 실체화**: 하드코딩된 'Pandas' 위주 컨텐츠를 `lectureId`에 기반한 동적 컨텐츠(환경 설정, 자료형 등)로 전환하여 실제 학습 동선 완성.
  4. **지시어 인식 오류 수정**: `"use client";` 표준 문법 적용을 통해 Turbopack 환경에서의 빌드 안정성 확보.
  
  ### 의사결정 9 — 최신 기술 스택 규격 준수 채택
  - **채택**: `params` 비동기 언래핑(`use()` hook) 및 동적 컨텐츠 맵핑 구조
  - **채택 이유**:
    - 프레임워크 사양을 준수하지 않은 동기식 접근은 런타임 ReferenceError를 유발함.
    - 강의 ID와 본문 데이터를 1:1 매칭함으로써 사용자가 "내가 선택한 강의를 정확히 배운다"는 신뢰를 제공.
    - 모델명 고정을 통해 API 호출의 예측 가능성 및 생성 품질 유지.
  
  > **핵심 진전**: 기술적 부채(Old API 사용)를 청산함과 동시에, 기획된 커리큘럼이 실제 학습 컨텐츠로 연결되는 "데이터의 실질적 흐름"을 완성함.

## Phase 10 — 전체 학습 플로우 고도화 및 교수자 권한 강화

### 배경
> "선생님이 강좌 이름과 목차를 직접 관리하고, AI가 학생 개개인의 수준에 맞춘 문제를 생성할 수 있도록 가이드라인을 입력하는 구조로 시스템 고도화"

### 주요 변경 사항
1. **전체 학습 Flow 재정의**:
   - **선생님**: 강좌/목차 생성 → 강의 내용 입력 → AI 집중 영역(프롬프트 가이드) 입력.
   - **학생**: 학습자의 현재 수준에 맞춰 AI가 개인화된 문제(퀴즈/코테)를 동적으로 생성하여 제공.
2. **교수자 편집 도구 개선**:
   - `teacher/series/[id]/edit` 페이지에서 강좌(Series)의 이름을 즉시 수정할 수 있는 기능 추가.
3. **콘텐츠 관리 전략 단순화**:
   - 기존의 개별 문제 관리 방식에서 벗어나, 퀴즈와 코딩 테스트의 최소/최대 개수만 설정하는 정책 형식으로 간소화. 나머지는 선생님이 입력한 가이드를 바탕으로 AI가 생성.

### 의사결정 10 — 교수자 주도 AI 개인화 학습 채택
- **채택**: 강좌명 수정 기능 및 퀴즈/코테 수량 베이스 설정 UI
- **채택 이유**:
  - 강사가 모든 문제를 직접 만드는 부담을 줄이면서도, AI가 어떤 방향으로 문제를 내야 할지 통제권을 가짐.
  - 학생은 획일화된 문제가 아닌 본인의 취약점과 수준에 맞는 문제를 풀게 되어 학습 효율 극대화.

> **핵심 성과**: "선생님의 전문성"과 "AI의 개인화 생성 능력"이 결합된 하이브리드 교육 모델 확립.
